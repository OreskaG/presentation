DRY

Don't Repeat Yourself

DRY is a principle of software development aimed at reducing repetition of software patterns, replacing it with abstractions or using data normalization to avoid redundancy.

History

The principle has been formulated by Andy Hunt and Dave Thomas in their book The Pragmatic Programmer (1999). They apply it quite broadly to include database schemas, test plans, the build system, even documentation

KISS

Keep It Simple Stupid

KISS principle states that most systems work best if they are kept simple rather than made complicated, therefore, simplicity should be a key goal in design, and unnecessary complexity should be avoided

History

The acronym was reportedly coined by Kelly Johnson, lead engineer at the Lockheed Skunk Works (creators of the Lockheed U-2 and SR-71 Blackbird spy planes, among many others). However, the variant "Keep it Short and Simple" is attested from a 1938 issue of the Minneapolis Star.

YAGNI

You Aren't Gonna Need It

YAGNI, the process and principle of software design, in which the rejection of redundant functionality is declared as the main goal and/or value, that is, the refusal to add functionality that is not directly needed.

History

YAGNI was the main slogan of the book Extreme Programming Installed (2001)

One of the authors Ron Jeffries has written: "Always implement things when you actually need them, never when you just foresee that you need them."

  • The time that would have been spent on adding, testing and improving the necessary functionality is wasted.
  • New functions should be debugged, documented and maintained.
  • The new functionality limits what can be done in the future â€” unnecessary new features may prevent you from adding new ones later.
  • Until the new features are really needed, it's hard to fully anticipate what they should do and test them. If the new features are not thoroughly tested, they may not work properly when needed later.
  • This leads to the fact that the software becomes more complex (sometimes excessively complex).
  • If all the functionality is not documented, it may remain unknown to users, but it can create various risks for the security of the user system.
  • Adding new functionality can lead to the desire for even more new functionality, resulting in a "snowball" effect.
